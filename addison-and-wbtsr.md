# Addison and WBTSR

Addison is the system. WBTSR is the workflow used to build it.

That distinction matters because the two are related, but they are not interchangeable.

## Addison

Addison is the built system.

It is a local, daemon-backed state system centered on persistent artifacts, deterministic context construction, stateless inference, and human-governed operation.

When this repository talks about daemon authority, artifact-backed memory, runtime boundaries, or public proof posture, it is talking about Addison.

## WBTSR

WBTSR is the named workflow used to structure the work around building Addison.

WBTSR stands for **Work, Build, Truth, Session, Rehydrate**.

It is a human-governed method built around:

- bounded work slices
- explicit truth surfaces
- separate production and review roles
- disciplined closeout
- restart-safe rehydration

WBTSR emerged from the Addison build process, but it is the workflow, not the system itself.

## Why The Distinction Matters

Without this distinction, it becomes easy to blur together:

- the thing being built
- the method used to build it
- the evidence that the method worked in one real context

Addison is evidence that the workflow was used in practice. WBTSR is the more portable method-level result.

That does not make them identical.

## Simple Version

- **Addison** = the system
- **WBTSR** = the workflow used to build it
- **methodology** = the general category term for that workflow posture

This public repo is primarily about Addison as a system, while still preserving the distinction that WBTSR is the named workflow extracted from building it.
