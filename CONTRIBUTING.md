# Contributing

## Scope

Thank you for your interest in Addison.

This public repository exists as a documentation, architecture, and public-proof surface for the Addison project. Its purpose is to help outside readers understand what the system is, how it is being shaped, what has been proven so far, and where the public boundary is drawn.

It is not the full private development repository, and it should not be treated as a request surface for exposing internal artifacts, private cognition layers, or live operator-linked runtime materials.

Because of that, contributions should help improve public understanding, clarity, correctness, and boundary discipline.

---

## What Kinds of Contributions Are Useful

Useful contributions are those that strengthen the public-facing quality of the repository without pushing it beyond its intended scope.

Examples include:

- fixing factual errors, broken links, typos, or unclear wording
- improving architecture explanations for external readers
- refining documentation structure and readability
- suggesting clearer proof framing where public claims need tighter wording
- improving examples, templates, or explanatory diagrams
- identifying places where a document overclaims or underexplains
- helping keep terminology consistent across public docs

In general, the best contributions make the repository more truthful, more legible, and easier to understand.

---

## What Does Not Belong Here

This repository is not the place for contributions that assume the public repo should become a full mirror of the private system.

Examples of contributions that do not fit this repo well include:

- requests to publish private prompts, private memory artifacts, or internal cognition files
- pull requests that attempt to expose operator-linked internal materials
- changes that blur the boundary between public-safe examples and live private artifacts
- speculative implementation changes presented as though they are already proven
- attempts to turn the public repo into the main internal development surface
- contributions that pressure the project to disclose more than it is ready to disclose safely

A contribution can be technically interesting and still be out of scope for this repository.

---

## Documentation-First Expectation

Addison is being developed with a strong documentation-first posture.

That means public-facing changes should usually improve one or more of these areas:

- clarity
- truthfulness
- structure
- proof quality
- boundary discipline

If you propose a meaningful public-facing change, explain what problem it solves and why the change improves the repository as a public understanding surface.

The best contributions do not just add text. They improve the repo’s ability to say the right thing clearly and defensibly.

---

## Public Truthfulness Standard

Please do not strengthen claims beyond what the repository can actually support.

This project is intentionally trying to avoid inflated language, vague futurism, or claims that outrun evidence.

Good contributions should help preserve a few basic standards:

- say what is real
- distinguish observed proof from future direction
- avoid implying completeness where work is still partial
- avoid collapsing private development reality into public simplification
- keep public explanation grounded in what can be defended

If a sentence sounds stronger than the available public evidence, it should probably be tightened.

---

## Respect the Public/Private Boundary

A central rule of this repository is that public explanation and private operation are not the same thing.

Contributors should respect that boundary.

That means:

- public-safe examples should remain examples
- templates should not become leaks of real private artifacts
- proof docs should stay bounded and truthful
- architecture docs should explain without exposing private operational surfaces
- public clarity should not come at the cost of privacy discipline

When in doubt, prefer explanation over exposure.

---

## Issues and Pull Requests

If you open an issue or pull request, try to keep it specific.

Helpful issue types include:

- documentation correction
- terminology inconsistency
- confusing architecture explanation
- unclear boundary language
- proof wording that needs tightening
- missing public-facing explanation that can be added safely

Helpful pull requests are usually small to medium in scope and easy to review in isolation.

Large changes that touch many docs at once are harder to validate against the repository’s public truthfulness standard, so narrower changes are generally better.

---

## Style Expectations

Contributions should aim for writing that is:

- clear
- direct
- bounded
- readable by an external audience
- accurate about current project reality

Try to avoid:

- hype language
- unnecessary marketing tone
- overstated certainty
- vague abstraction without explanation
- making the public repo sound more complete than it is

The public surface should feel deliberate, legible, and trustworthy.

---

## Before Submitting a Change

Before submitting a contribution, check whether the change:

- improves public understanding
- stays within the public scope of the repo
- preserves privacy and boundary discipline
- avoids overstating what has been proven
- makes the documentation clearer rather than noisier

That is the basic test.

---

## Maintainer Direction

The maintainer may decline contributions that are well-intentioned but not aligned with the role of this repository.

That is not a rejection of outside interest. It is a boundary decision.

This repo is intentionally curated as a public-facing explanatory surface, not an everything-repo.

Changes that improve that surface are welcome.
Changes that push against the repo’s boundary may be declined even if they are thoughtful or technically strong.

---

## Practical Summary

Contribute here if your change helps Addison become easier to understand publicly, more truthful in its claims, clearer in its architecture and proof surfaces, or stronger in its boundary discipline.

Do not contribute here on the assumption that the goal is full disclosure of the private system.

This repository is meant to explain the project well in public, not to expose every part of how it operates privately.
