# Contributing

## Scope

Thank you for your interest in Addison.

This public repository is a curated documentation and proof surface. It exists to help outside readers understand what the system is, what the public repo currently supports, and where the public boundary is drawn.

It is not the full private development repository.

Contributions should improve public understanding, clarity, correctness, and boundary discipline.

---

## Read First

Before making documentation changes, read these files first:

1. [README.md](README.md)
2. [ARCHITECTURE.md](ARCHITECTURE.md)
3. [PUBLIC_STATUS.md](PUBLIC_STATUS.md)
4. [addison-and-wbtsr.md](addison-and-wbtsr.md)

These files define the public-facing explanation, current public claim boundary, and system/workflow distinction that new changes should preserve.

---

## What Helps

Useful contributions strengthen the public-facing quality of the repository without pushing it beyond its intended role. Examples include:

- fixing factual errors, broken links, typos, or unclear wording
- improving architecture explanations for outside readers
- identifying places where wording overclaims or underexplains
- tightening proof framing where public claims need narrower language
- helping keep terminology consistent across public docs
- improving navigation between existing public-facing surfaces

The best contributions make the repo more truthful, more legible, and easier to understand.

---

## What Does Not Belong

This repository is not the place for contributions that assume the public repo should mirror the private system. Examples include:

- requests to publish private prompts, memory artifacts, or internal cognition files
- changes that blur the boundary between public-safe explanation and live private artifacts
- speculative implementation claims presented as though they are already proven publicly
- contributions that pressure the project to disclose more than it is ready to disclose safely
- changes that collapse architecture explanation, public proof posture, and contributor guidance into one document role

A contribution can be technically interesting and still be out of scope for this repository. When in doubt, prefer explanation over exposure.

---

## Standards

This project is intentionally trying to avoid inflated language, vague futurism, and claims that outrun evidence. Contributions should meet a few basic standards:

- say what is real
- distinguish observed public support from future direction
- do not strengthen claims beyond what the repository can actually support
- write clearly and directly for an outside reader
- avoid hype language, marketing tone, or overstated certainty
- keep internal links accurate
- preserve document roles instead of quietly widening them — navigation docs, architecture docs, proof docs, boundary docs, and contributor guidance serve different purposes and should not be collapsed together

If a sentence sounds stronger than the available public surface can support, it should probably be tightened.

---

## Issues and Pull Requests

Keep issues and pull requests specific.

Helpful submissions usually cover a documentation correction, a terminology inconsistency, a confusing explanation, unclear boundary language, link repair, or proof wording that needs tightening.

Smaller, narrower changes are generally easier to review and more likely to land. Large changes that touch many docs at once are harder to validate against the repo's truthfulness standard.

---

## Maintainer Direction

The maintainer may decline contributions that are thoughtful but not aligned with the role of this repository. That is a boundary decision, not a dismissal of outside interest.

Changes that improve the public surface are welcome. Changes that push against the repo's boundary may be declined even when they are technically strong.
