# Contributing

## Scope

Thank you for your interest in Addison.

This public repository is a curated documentation and proof surface. It exists to help outside readers understand what the system is, what has been proven so far, and where the public boundary is drawn. It is not the full private development repository.

Contributions should improve public understanding, clarity, correctness, and boundary discipline.

---

## What's Useful

Useful contributions strengthen the public-facing quality of the repository without pushing it beyond its intended scope. Examples include:

- Fixing factual errors, broken links, typos, or unclear wording.
- Improving architecture or proof explanations for external readers.
- Identifying places where a document overclaims or underexplains.
- Suggesting clearer proof framing where public claims need tighter wording.
- Helping keep terminology consistent across public docs.

The best contributions make the repository more truthful, more legible, and easier to understand.

---

## What Doesn't Belong

This repository is not the place for contributions that assume the public repo should mirror the private system. Examples include:

- Requests to publish private prompts, memory artifacts, or internal cognition files.
- Changes that blur the boundary between public-safe examples and live private artifacts.
- Speculative implementation changes presented as though they are already proven.
- Contributions that pressure the project to disclose more than it is ready to disclose safely.

A contribution can be technically interesting and still be out of scope for this repository. When in doubt, prefer explanation over exposure.

---

## Standards

This project is intentionally trying to avoid inflated language, vague futurism, and claims that outrun evidence. Contributions should meet a few basic standards:

- Say what is real. Distinguish observed proof from future direction.
- Don't strengthen claims beyond what the repository can actually support.
- Write clearly and directly for an external audience.
- Avoid hype language, unnecessary marketing tone, or overstated certainty.

If a sentence sounds stronger than the available public evidence, it should probably be tightened.

---

## Issues and Pull Requests

Keep issues and pull requests specific. Helpful submissions might cover a documentation correction, a terminology inconsistency, a confusing explanation, unclear boundary language, or proof wording that needs tightening.

Smaller, narrower changes are generally easier to review and more likely to land. Large changes that touch many docs at once are harder to validate against the repository's truthfulness standard.

---

## Maintainer Direction

The maintainer may decline contributions that are well-intentioned but not aligned with the role of this repository. That is not a rejection of outside interest — it is a boundary decision.

Changes that improve the public surface are welcome. Changes that push against the repo's boundary may be declined even if they are thoughtful or technically strong.
